
# 🏗 АРХИТЕКТУРНЫЙ ПЛАН: "SYSTEMS THAT SHIP"

Мы используем паттерн, близкий к **Feature-Sliced Design (FSD)**, но адаптированный под реалии Next.js App Router, чтобы не переусложнять. Главное правило: **Разделение ответственности (Separation of Concerns).**

## 1. ГЛОБАЛЬНЫЙ СТЕК И ИНСТРУМЕНТЫ (Ядро)
*   **Фреймворк:** Next.js 14+ (App Router). 90% компонентов — серверные (RSC).
*   **Язык:** TypeScript (Strict mode: ON). Никаких `any`. Никогда.
*   **Стилизация:** Tailwind CSS + `clsx` + `tailwind-merge` (чтобы красиво комбинировать классы).
*   **Валидация данных:** Zod (проверяем всё, что летит с фронта в БД).
*   **Управление состоянием:** Забудь про Redux/MobX. Для блога нам хватит URL-параметров (для пагинации/поиска), Server State (кэш Next.js) и React Context (для состояния кастомного курсора или Lenis-скролла).

## 2. СТРУКТУРА ДИРЕКТОРИЙ (Скелет проекта)

Всё держим в папке `src/`. Корень должен быть чистым.

```text
src/
├── app/                  # ТОЛЬКО роутинг, Layouts и Pages
│   ├── (public)/         # Группа публичных страниц (без влияния на URL)
│   │   ├── page.tsx      # Главная (список постов)
│   │   ├── posts/[slug]/ # Страница конкретного поста
│   │   └── layout.tsx    # Паблик-лейаут (Навбар, Футер, Lenis Scroll)
│   ├── (admin)/          # Группа админки
│   │   ├── login/        # Вход
│   │   ├── dashboard/    # Админка
│   │   └── layout.tsx    # Админ-лейаут (Сайдбар, проверка авторизации)
│   ├── api/              # Webhooks (если понадобятся, но в основном юзаем Actions)
│   └── globals.css       # Глобальные стили, переменные
│
├── components/           # UI-слой (кирпичики)
│   ├── ui/               # Глупые (dumb) компоненты: Button, Input, Modal (Shadcn UI style)
│   ├── blocks/           # Смарт-компоненты: PostCard, HeroSection, AdminEditor
│   └── layout/           # Header, Footer, CustomCursor
│
├── lib/                  # Утилиты и конфигурации
│   ├── utils.ts          # Функция cn() для Tailwind, форматтеры дат
│   ├── supabase/         # Инициализация клиента Supabase (server + browser)
│   └── constants.ts      # Глобальные переменные (SEO-дефолты, лимиты пагинации)
│
├── actions/              # NEXT.JS SERVER ACTIONS (Мутации)
│   ├── auth.ts           # Логин, логаут
│   └── posts.ts          # Создать, обновить, удалить пост (server-side only)
│
├── queries/              # ФЕТЧИНГ ДАННЫХ (Чтение)
│   └── posts.ts          # getPosts(), getPostBySlug() (для использования в серверных компонентах)
│
└── types/                # ГЛОБАЛЬНЫЕ ТИПЫ
    ├── supabase.ts       # Сгенерированные типы из БД
    └── index.ts          # Кастомные интерфейсы (Props)
```

**Почему так, а не иначе?** 
*   **Сеньорская фишка:** Если я захочу посмотреть, как ты получаешь данные, я пойду в `queries/`. Если захочу посмотреть, как ты их меняешь — пойду в `actions/`. Если захочу посмотреть UI — в `components/`. Я не буду искать логику размазанной по файлам `page.tsx`.

## 3. СТРАТЕГИЯ РЕНДЕРИНГА (Движок)

Это то, на чем сыплются 80% джунов. Мы делаем элитно:

*   **Публичная часть (Главная, Пост): SSR + ISR.** 
    *   Страницы собираются на сервере. Никаких лоадеров на старте.
    *   Данные берем напрямую в серверном компоненте (RSC).
    *   *Магия:* Когда ты в админке нажимаешь "Опубликовать", срабатывает Server Action, который делает `revalidatePath('/')`. Next.js сбрасывает кэш, и новые посетители мгновенно видят новый пост. Это дает скорость статики + актуальность динамики.
*   **Админка:** Клиентская магия.
    *   В админке SEO не нужно. Нам нужна интерактивность. Редактор (TipTap) будет клиентским компонентом (`"use client"`), чтобы быстро реагировать на нажатия клавиш.

## 4. БЕЗОПАСНОСТЬ И МАРШРУТИЗАЦИЯ (Фейс-контроль)

Никаких проверок через `useEffect` на клиенте! Это дыра в безопасности (пользователь на секунду увидит админку, прежде чем его выкинет).

*   **Middleware (`src/middleware.ts`):** 
    Мы ставим охранника на уровне Edge-функций Next.js.
    1. Юзер стучится на `/dashboard`.
    2. Middleware проверяет куки Supabase Auth до того, как страница начнет рендериться.
    3. Нет токена? Мгновенный 307 Redirect на `/login`.
    4. Есть токен, но стучится на `/login`? Redirect на `/dashboard`.

## 5. ПОТОК ДАННЫХ: КАК РОЖДАЕТСЯ ПОСТ (Data Flow)

Чтобы ты четко понимал, как данные ходят туда-сюда:

1.  **Админка (Client):** Ты вводишь заголовок и текст в форму. Картинку грузишь через инпут.
2.  **Действие (Action):** Форма вызывает Server Action `createPost(formData)`.
3.  **Валидация (Server):** В экшене `Zod` проверяет: "А заголовок не пустой? А slug валидный?".
4.  **Storage (Server):** Если есть картинка, экшен кидает её в Supabase Storage, получает URL.
5.  **База (Server):** Экшен пишет строку в БД (title, content, image_url).
6.  **Ревалидация (Server):** Экшен вызывает `revalidatePath('/')`.
7.  **Ответ:** Клиент получает `{ success: true }`, форма очищается, всплывает красивая Toast-нотификация.

## 6. АРХИТЕКТУРА ИНТЕРАКТИВА (Подготовка к AWWWARDS)

Раз мы целимся в дизайн, архитектура должна его поддерживать:

*   **Изоляция Client Components:** GSAP, Lenis Scroll и кастомный курсор требуют доступа к `window` и DOM. Мы не вешаем `"use client"` на целые страницы. Мы создаем компоненты-обертки (например, `<SmoothScrollProvider>`), которые оборачивают серверный контент. 
*   **Performance:** Шрифт грузим только через `next/font` (забудем про Google Fonts link в HTML). Изображения — строго через `<Image />` с заранее заданными пропорциями (aspect-ratio), чтобы не было CLS (Cumulative Layout Shift).
